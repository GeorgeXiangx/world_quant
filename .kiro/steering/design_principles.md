# 软件设计原则速查

编码时应主动对照以下原则审视代码质量，避免过度设计和技术债务。

## SOLID 原则

| 原则 | 含义 | 违反信号 |
|---|---|---|
| S - 单一职责（SRP） | 一个类只做一件事，只有一个修改理由 | 类超过300行、方法超过50行、类名含"And"或"Manager" |
| O - 开闭原则（OCP） | 对扩展开放，对修改关闭 | 新增功能需要改已有代码的 if-else |
| L - 里氏替换（LSP） | 子类可以替换父类而不影响程序正确性 | 子类重写方法后抛出父类未声明的异常 |
| I - 接口隔离（ISP） | 接口应该小而专，不强迫实现不需要的方法 | 实现类中有空方法或抛 UnsupportedOperationException |
| D - 依赖倒置（DIP） | 高层模块不依赖低层模块，都依赖抽象 | Controller 直接 new ServiceImpl |

## 通用设计原则

| 原则 | 含义 | 实践要点 |
|---|---|---|
| KISS（Keep It Simple, Stupid） | 保持简单，避免不必要的复杂性 | 能用简单 if 解决的不要用设计模式；优先选择可读性高的写法 |
| DRY（Don't Repeat Yourself） | 不要重复自己 | 相同逻辑出现2次以上就抽取公共方法；如 `InvokeResp.toEntity()` |
| YAGNI（You Aren't Gonna Need It） | 不要提前实现用不到的功能 | 不写"以后可能用到"的代码；需求驱动开发 |
| LoD（迪米特法则/最少知识原则） | 一个对象应尽量少了解其他对象的内部 | 避免链式调用 `a.getB().getC().doSomething()`；用封装方法代替 |
| CQS（命令查询分离） | 方法要么改变状态（命令），要么返回数据（查询），不要两者兼做 | `save()` 不返回查询结果；查询方法不产生副作用 |
| 组合优于继承 | 优先使用组合/委托而非继承来复用代码 | 继承层级不超过3层；用接口+组合替代深层继承 |
| 面向接口编程 | 依赖接口而非具体实现 | Service 层接口与 Impl 分离；方法参数用 List 而非 ArrayList |
| 防御性编程 | 不信任外部输入，做好边界检查 | 参数校验、null 检查、异常处理 |
| 快速失败（Fail Fast） | 尽早发现错误并抛出，不要让错误数据继续传播 | 方法入口做参数校验；`Objects.requireNonNull()` |

## 编码时的自检清单

- 这个方法是否只做了一件事？→ **SRP**
- 新增功能是否需要修改已有代码？→ **OCP**，考虑用策略/模板方法扩展
- 这段逻辑是否在别处也写过？→ **DRY**，抽取公共方法
- 这个功能现在真的需要吗？→ **YAGNI**，不要过度设计
- 这段代码能不能更简单？→ **KISS**，简单优于巧妙
- 这个类是否知道了太多其他类的细节？→ **LoD**，考虑封装
- 入参是否做了校验？null 是否处理了？→ **防御性编程**
- 异常是否在合适的层级处理？→ **快速失败**
